## 분산 시스템 안정성 (Resilience, 장애 격리·복원력)

### 정의
분산 시스템에서는 여러 서비스가 네트워크로 서로를 호출하기 때문에, 어느 한순간에 모든 서비스가 완벽하게 살아있을 것이라는 기대 자체가 비현실적이다. 일부 서비스가 잠깐 죽거나(다운), 평소보다 느리게 응답하는(지연) 상황은 예외가 아니라 일상이다. 그래서 Resilience(복원력)는 "장애를 어떻게 없앨까"가 아니라 "부분 장애가 발생하는 것은 당연하다는 전제 아래, 그 장애가 전체 시스템으로 전파되는 것을 어떻게 막고 빠르게 회복할까"를 다루는 개념이다.

집의 전기 배선에 빗대면 이해가 쉽다. 방 하나에서 누전이 발생했을 때 두꺼비집(차단기, Circuit Breaker)이 그 회로만 똑 떨어뜨려 차단해 주기 때문에 집 전체가 정전되지 않는다. Resilience는 바로 이 두꺼비집과 같다. 문제가 생긴 부분만 격리(isolation)하고, 나머지는 계속 살아있게 하며, 시간이 지나면 다시 회로를 붙여보는 것이다.

### 등장 배경 / 왜 필요한가
모놀리식(monolith) 시절에는 모든 기능이 한 프로세스 안에 있어서 "내부 함수 호출"이 곧 서비스 간 통신이었다. 함수 호출은 거의 즉시 끝나고 실패하지도 않는다. 하지만 [MSA](./msa.md) 구조로 넘어오면서 서비스 A가 서비스 B를, B가 다시 C를 네트워크로 호출하는 긴 호출 체인이 생겼다. 네트워크 호출은 함수 호출과 달리 느려질 수도, 끊길 수도, 무한정 응답을 안 줄 수도 있다.

문제는 한 서비스의 지연이 호출한 쪽의 스레드·커넥션 자원을 붙잡아 둔다는 점이다. C가 느려지면 → C를 기다리는 B의 스레드가 묶이고 → B가 느려지면 → B를 기다리는 A의 스레드까지 모두 묶인다. 이렇게 말단의 작은 지연이 호출 체인을 거꾸로 타고 올라가 시스템 전체를 마비시키는 현상을 연쇄 장애(cascading failure)라고 한다.

```
연쇄 장애(cascading failure)

[정상]
사용자 -> A(빠름) -> B(빠름) -> C(빠름)            전체 정상

[C 지연 발생]
사용자 -> A(대기) -> B(대기) -> C(느림/응답없음)
            ↑          ↑          ↑
      스레드 묶임   스레드 묶임   원인 발생
   (자원 고갈이 A까지 역류 -> A도 다운 -> 전체 장애)
```

말단 서비스 하나의 지연이 "기다림"이라는 형태로 상류 서비스의 자원을 잠식하기 때문에, 장애 전파를 끊어줄 안전장치가 필요하다. 그 안전장치들의 모음이 곧 Resilience 패턴이다.

### 핵심 개념 / 패턴

#### Circuit Breaker (서킷 브레이커)
- 정의: 특정 서비스 호출의 실패율을 계속 감시하다가, 실패가 임계값을 넘으면 회로를 "열어" 그 서비스로의 호출 자체를 한동안 차단하는 패턴이다. 두꺼비집과 동일한 발상이다.
- 동작: 정상 상태(Closed)에서는 호출을 그대로 통과시키되 실패 횟수를 센다. 실패율이 임계치를 넘으면 열린 상태(Open)로 전환되어, 일정 시간 동안은 실제 호출을 시도조차 하지 않고 즉시 실패(fail fast)시킨다. 이렇게 하면 이미 죽은 서비스를 계속 두드려 자원을 낭비하는 것을 막는다. 대기 시간이 지나면 반열림 상태(Half-Open)로 바뀌어 시험삼아 소수의 요청만 흘려보내고, 성공하면 Closed로 복귀, 실패하면 다시 Open으로 돌아간다.

```
서킷 브레이커 상태 전이도

          실패율 임계값 초과
   ┌──────────────────────────────┐
   │                              ▼
┌────────┐                   ┌────────┐
│ CLOSED │                   │  OPEN  │
│(정상통과)│                   │(즉시차단)│
└────────┘                   └────────┘
   ▲   ▲                          │
   │   │                          │ 대기 시간 경과
   │   │ 시험 요청 성공             ▼
   │   │                  ┌──────────────┐
   │   └──────────────────│  HALF-OPEN   │
   │   (Closed 복귀)       │ (시험 요청만)  │
   │                      └──────────────┘
   │                              │
   └──────────────────────────────┘
              시험 요청 실패 (Open 복귀)
```

#### Retry (재시도) + Backoff
- 정의: 일시적인 실패(네트워크 순간 끊김, 잠깐의 과부하 등)는 다시 시도하면 성공할 수 있으므로, 실패한 호출을 정해진 횟수만큼 재시도하는 패턴이다.
- 동작: 그냥 곧바로 반복 호출하면 이미 힘든 서버를 더 때리게 되므로, 재시도 사이의 간격을 점점 늘리는 백오프(backoff)를 함께 쓴다. 보통 지수 백오프(exponential backoff)로 1초 → 2초 → 4초처럼 간격을 키우고, 여러 클라이언트가 동시에 같은 타이밍에 재시도해 부하가 한꺼번에 몰리는 것을 막기 위해 약간의 무작위 시간(jitter)을 더한다.

```
재시도 + 지수 백오프(+jitter)

호출 ──실패──> [대기 1초±α] ──재시도──> 실패
                                       │
              [대기 2초±α] <───────────┘
                  │
                  └──재시도──> 실패
                                │
              [대기 4초±α] <─────┘
                  │
                  └──재시도──> 성공 ✔ (또는 최대 횟수 초과 시 최종 실패)
```

#### Timeout (타임아웃)
- 정의: 응답을 무한정 기다리지 않고, 정해진 시간이 지나면 호출을 강제로 포기하고 실패로 처리하는 패턴이다.
- 동작: 연쇄 장애의 근본 원인은 "끝나지 않는 기다림"으로 인한 자원 점유다. 타임아웃을 걸어두면 느린 서비스를 기다리던 스레드를 정해진 시간 뒤에 회수할 수 있어, 자원이 무한정 묶이는 것을 막는다. 서킷 브레이커·재시도가 제대로 동작하려면 그 전제로 타임아웃이 반드시 설정되어 있어야 한다.

```
타임아웃 없음 vs 있음

[없음]  호출 ───────────────────────────────...> (영원히 대기)
        스레드가 회수되지 않음 -> 자원 고갈

[있음]  호출 ──────[ 2초 ]──X 타임아웃! 실패 처리
        스레드 즉시 회수 -> 폴백 또는 에러 반환
```

#### Bulkhead (격벽)
- 정의: 배의 선체를 여러 칸으로 나눠 한 칸에 구멍이 나도 배 전체가 침몰하지 않게 하는 격벽(bulkhead)에서 따온 이름으로, 자원(스레드 풀·커넥션 풀)을 호출 대상별로 분리해 한 서비스의 장애가 다른 호출의 자원까지 잠식하지 못하게 막는 패턴이다.
- 동작: 모든 외부 호출이 하나의 공용 스레드 풀을 공유하면, 느린 서비스 하나가 풀의 스레드를 전부 점유해 멀쩡한 서비스 호출까지 자원 부족으로 실패하게 만든다. 호출 대상마다 별도의 풀을 할당하면, B가 느려져도 그 영향은 "B 전용 풀"에만 갇히고 C·D 호출은 자기 풀로 정상 동작한다.

```
격벽(bulkhead) 자원 격리

[공용 풀 - 위험]                [격벽 적용 - 안전]
   ┌──────────────┐              B풀 ┌────┐ -> B(느림, 여기만 가득참)
   │  공용 스레드   │ -> B(느림)        └────┘
   │  풀이 B에 의해 │ -> C(대기X)    C풀 ┌────┐ -> C(정상)
   │  전부 잠식됨   │ -> D(대기X)        └────┘
   └──────────────┘              D풀 ┌────┐ -> D(정상)
   C, D까지 동반 실패                  └────┘  C, D는 영향 없음
```

#### Fallback (폴백)
- 정의: 호출이 실패하거나 서킷이 열려 있을 때, 에러를 그대로 사용자에게 던지는 대신 미리 준비한 대체 응답을 돌려주는 패턴이다.
- 동작: 추천 서비스가 죽었을 때 "맞춤 추천 목록"을 비우고 에러를 내는 대신, 캐시된 이전 추천이나 인기 상품 같은 무난한 기본값을 보여주는 식이다. 핵심 기능은 아니지만 없으면 화면이 깨지는 부가 기능을, 장애 상황에서도 우아하게 저하(graceful degradation)된 형태로 계속 제공하는 것이 목적이다.

```
폴백(fallback) 흐름

호출 ─성공─> 실제 응답 반환
   │
   └─실패/서킷 Open─> 폴백 실행
                       ├─> 캐시된 이전 값
                       ├─> 기본값(빈 목록, 인기 상품 등)
                       └─> 안내 메시지 ("잠시 후 다시 시도")
```

### 실무 예제
Ex: Spring Boot 환경에서 Resilience4j를 사용해 주문 서비스가 결제 서비스를 호출할 때 서킷 브레이커·타임아웃·재시도를 적용하는 예시이다.

먼저 `application.yml`에 서킷 브레이커 인스턴스를 정의한다. 실패율이 50%를 넘으면 회로를 열고, 10초 뒤 Half-Open으로 전환해 시험 요청 3건을 흘려보내도록 설정한다.

```yaml
resilience4j:
  circuitbreaker:
    instances:
      paymentService:
        sliding-window-type: COUNT_BASED      # 최근 N건 기준으로 실패율 집계
        sliding-window-size: 10               # 최근 10건을 관찰 창으로 사용
        minimum-number-of-calls: 5            # 최소 5건은 모여야 실패율 판정 시작
        failure-rate-threshold: 50            # 실패율 50% 초과 시 Open 전환
        wait-duration-in-open-state: 10s      # Open 유지 시간(이후 Half-Open)
        permitted-number-of-calls-in-half-open-state: 3  # Half-Open 시험 요청 수
        automatic-transition-from-open-to-half-open-enabled: true
  timelimiter:
    instances:
      paymentService:
        timeout-duration: 2s                  # 2초 안에 응답 없으면 타임아웃
  retry:
    instances:
      paymentService:
        max-attempts: 3                       # 최초 1회 + 재시도 2회
        wait-duration: 1s                     # 재시도 간 대기
        enable-exponential-backoff: true      # 1s -> 2s -> 4s 지수 백오프
        exponential-backoff-multiplier: 2
```

그다음 호출 메서드에 어노테이션을 붙인다. `fallbackMethod`에는 같은 시그니처에 예외 파라미터를 추가한 폴백 메서드를 지정한다.

```java
@Service
public class OrderService {

    private final PaymentClient paymentClient;

    public OrderService(PaymentClient paymentClient) {
        this.paymentClient = paymentClient;
    }

    // name 값("paymentService")이 application.yml의 인스턴스와 매핑된다.
    @CircuitBreaker(name = "paymentService", fallbackMethod = "paymentFallback")
    @Retry(name = "paymentService")
    @TimeLimiter(name = "paymentService")
    public CompletableFuture<PaymentResult> pay(Long orderId, long amount) {
        return CompletableFuture.supplyAsync(
            () -> paymentClient.requestPayment(orderId, amount)
        );
    }

    // 폴백: 원본 메서드 파라미터 + 마지막에 Throwable 추가
    private CompletableFuture<PaymentResult> paymentFallback(
            Long orderId, long amount, Throwable t) {
        // 결제 서비스 장애 시 "대기(PENDING)" 상태로 응답하고,
        // 나중에 비동기 재처리 큐로 넘겨 우아하게 저하시킨다.
        return CompletableFuture.completedFuture(
            PaymentResult.pending(orderId, "결제 서비스 지연 - 잠시 후 처리됩니다")
        );
    }
}
```

이렇게 하면 결제 서비스가 느려지거나 죽어도, 주문 서비스는 2초 만에 타임아웃 → (필요 시) 재시도 → 그래도 실패하면 서킷을 열고 즉시 폴백으로 PENDING 응답을 돌려준다. 주문 서비스의 스레드가 결제 서비스를 기다리다 고갈되는 일이 없어진다.

### 주의점 / 안티패턴
- 무한 재시도로 인한 부하 증폭(retry storm): 백오프 없이, 혹은 너무 많은 횟수로 재시도하면 이미 힘든 서버를 더 강하게 두드린다. 게다가 여러 클라이언트가 동시에 재시도하면 트래픽이 폭증해 멀쩡하던 서버까지 무너뜨린다. 재시도는 짧은 횟수 + 지수 백오프 + jitter를 반드시 함께 쓰고, 재시도해도 의미 없는 영구적 실패(4xx 등)에는 재시도하지 않는다.
- 타임아웃 미설정: 가장 흔하고 치명적인 실수다. 타임아웃이 없으면 느린 서비스를 무한정 기다리며 스레드가 묶이고, 결국 연쇄 장애의 출발점이 된다. 서킷 브레이커나 재시도를 아무리 잘 걸어도 타임아웃이 없으면 무용지물이다. 모든 외부 호출에는 타임아웃을 명시한다.
- 서킷 임계값을 잘못 잡기: 실패율 임계값이 너무 낮거나 최소 호출 수가 너무 작으면, 일시적인 한두 건 실패에도 서킷이 열려 정상 트래픽까지 차단하는 과민 반응이 발생한다. 반대로 임계값이 너무 높거나 관찰 창이 지나치게 크면, 장애가 명백한데도 서킷이 열리지 않아 보호 효과가 사라진다. 서비스별 평소 실패율과 트래픽량을 기준으로 임계값을 정하고 모니터링으로 조정해야 한다.
- 재시도와 타임아웃·서킷의 시간 예산을 따로 노는 채로 두기: 상류의 타임아웃은 (단일 호출 타임아웃 × 재시도 횟수)보다 커야 한다. 그렇지 않으면 하류가 재시도를 다 끝내기도 전에 상류가 먼저 타임아웃 나서, 재시도 설정이 무의미해지고 중복 호출만 남는다.
- 폴백이 또 다른 외부 호출에 의존하기: 폴백 로직 안에서 다시 네트워크 호출을 하면, 그 호출마저 실패할 때 폴백 자체가 장애 지점이 된다. 폴백은 캐시·기본값처럼 외부 의존이 적고 빠르게 끝나는 경로로 구성한다.
