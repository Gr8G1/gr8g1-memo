## 분산 데이터 일관성 (Distributed Data Consistency)

### 정의
분산 데이터 일관성이란, 여러 개의 서비스가 각자 자기 데이터베이스를 따로 가지고 있는 환경(MSA, Microservices Architecture)에서 "여러 DB에 걸친 변경을 어떻게 모순 없이 맞출 것인가"를 다루는 문제이자 그 해법들의 모음이다.
[MSA](./msa.md)에서는 보통 서비스마다 DB를 분리하는 Database per Service 원칙을 따른다. 그 결과 주문 DB, 결제 DB, 재고 DB가 물리적으로 따로 존재하고, 이 셋을 하나의 ACID 트랜잭션(원자성·일관성·격리성·지속성을 보장하는 단일 작업 단위)으로 묶어 한 번에 커밋/롤백할 수 없게 된다.

- Ex: 한 식당의 주방장 한 명이 모든 요리를 도맡으면, 손님이 주문을 취소했을 때 그가 만들던 음식을 곧바로 멈추면 된다(단일 트랜잭션). 하지만 전채·메인·디저트를 서로 다른 매장의 요리사 세 명이 따로 만든다면(서비스마다 분리된 DB), "메인 재료가 떨어졌으니 전채와 디저트도 전부 취소해 주세요"라는 말을 일일이 전화로 전해 가며 맞춰야 한다. 모두를 한 번에 멈출 스위치가 없는 상태, 그것이 분산 데이터 일관성 문제다.

즉, 모놀리식(Monolith)에서는 DB 한 곳에서 `BEGIN ... COMMIT` 한 줄로 끝나던 강한 일관성이, MSA에서는 더 이상 공짜로 주어지지 않는다. 그래서 "조금 늦더라도 결국엔 맞춰진다"는 새로운 약속(최종 일관성)과, 그 약속을 구현하는 패턴들이 필요해졌다.

### 등장 배경 / 왜 필요한가
여러 DB를 하나처럼 묶으려는 고전적 시도가 바로 2PC(Two-Phase Commit, 2단계 커밋)다. 중앙의 코디네이터(Coordinator)가 모든 참여자에게 "준비됐나?"를 묻고(Prepare 단계), 전원이 OK하면 "커밋해"를 명령(Commit 단계)하는 방식이다. 이론적으로는 여러 DB에 걸친 원자성을 보장하지만, 실무에서는 다음 한계가 크다.

- 블로킹(Blocking): Prepare 후 Commit이 올 때까지 모든 참여자가 자원을 잠그고 대기한다. 코디네이터가 그 사이에 죽으면 참여자들은 풀지도 커밋하지도 못한 채 묶인다.
- 가용성 저하: 참여자가 한 곳이라도 응답하지 않으면 전체가 멈춘다. 서비스가 많아질수록 "한 곳의 느림/장애"가 전체를 멈추게 할 확률이 곱셈으로 커진다.
- 확장성 부족: 동기적으로 전원의 합의를 기다리는 구조라 서비스 수와 트래픽이 늘수록 병목이 된다.

여기에 분산 시스템의 근본 한계인 CAP 정리(CAP Theorem)가 겹친다. CAP은 네트워크 분단(Partition)이 발생했을 때 일관성(Consistency)과 가용성(Availability)을 동시에 완벽히 만족할 수 없으니 둘 중 하나를 양보해야 한다는 정리다. 네트워크 단절은 분산 환경에서 피할 수 없으므로(P는 사실상 필수), 대부분의 MSA는 "잠깐의 불일치를 허용하더라도 서비스는 계속 살아 있게 한다"는 가용성 쪽을 택한다.

그 선택의 결과가 최종 일관성(Eventual Consistency)이다. "모든 시점에 항상 똑같다"가 아니라, "새로운 변경이 더 들어오지 않는다면 일정 시간 뒤에는 모든 복제본/서비스가 같은 상태로 수렴한다"는 약한 보장이다. 강한 일관성(Strong Consistency)을 포기하는 대신, 시스템 전체가 멈추지 않고 부분 장애를 견디며 확장할 수 있는 길을 얻는 것이다.

### 핵심 개념 / 패턴

#### Saga 패턴
Saga는 "하나의 큰 분산 트랜잭션"을 "여러 개의 로컬 트랜잭션(각 서비스가 자기 DB 안에서만 처리하는 작은 트랜잭션)의 연속"으로 쪼갠 것이다. 각 단계가 끝날 때마다 그 결과를 다음 단계로 전달하고, 중간에 실패하면 이미 성공한 단계들을 되돌리는 보상 트랜잭션을 역순으로 실행한다. Saga를 진행하는 방식은 누가 흐름을 주도하느냐에 따라 크게 두 가지로 나뉜다.

**1) Choreography(코레오그래피, 이벤트 기반)**

중앙 지휘자가 없고, 각 서비스가 자기 일을 끝낸 뒤 이벤트를 발행(publish)하면 다음 서비스가 그 이벤트를 구독(subscribe)해 알아서 다음 일을 이어 받는 방식이다. ([이벤트 기반](./event-driven.md) 구조 위에서 동작한다.) 마치 댄서들이 지휘자 없이 음악(이벤트)에 맞춰 자기 차례에 춤을 추는 군무(群舞)와 같다.

```
[주문 서비스]                [결제 서비스]                [재고 서비스]
     │                            │                           │
     │  주문 생성(로컬 TX)          │                           │
     ├── OrderCreated 이벤트 ──────▶                           │
     │                            │  결제 처리(로컬 TX)          │
     │                            ├── PaymentCompleted 이벤트 ──▶
     │                            │                           │  재고 차감(로컬 TX)
     │                            │                           │
     │           (각 서비스가 다음 이벤트를 듣고 스스로 이어받음)
```

**2) Orchestration(오케스트레이션, 중앙 조정자)**

별도의 오케스트레이터(Orchestrator, 조정자/Saga 매니저)가 전체 흐름을 손에 쥐고, "이제 결제해", "이제 재고 차감해"라고 각 서비스에 직접 명령(command)을 보내고 응답을 받아 다음 단계를 결정한다. 지휘자 한 명이 단원들에게 차례로 신호를 주는 오케스트라 연주와 같다.

```
              ┌──────────────────────────────┐
              │   Orchestrator (Saga 조정자)   │
              └──────────────────────────────┘
                 │  ▲      │  ▲       │  ▲
        ① 결제 명령│  │성공  ② 차감명령│ │성공  (실패 시 ③ 보상 명령)
                 ▼  │      ▼  │       ▼  │
            [결제 서비스]   [재고 서비스]   [주문 서비스]
                 (각 서비스는 명령을 받고 자기 로컬 TX만 수행)
```

**Choreography vs Orchestration 비교**

| 관점 | Choreography(이벤트 기반) | Orchestration(중앙 조정자) |
| --- | --- | --- |
| 흐름 제어 | 분산(각 서비스가 이벤트 듣고 자율 결정) | 중앙(Orchestrator가 명령으로 통제) |
| 결합도 | 서비스 간 느슨한 결합, 서로를 직접 모름 | 조정자가 모든 서비스를 알아 결합도 다소 높음 |
| 흐름 파악 | 흐름이 코드 곳곳에 흩어져 전체 추적이 어려움 | 흐름이 한 곳에 모여 가독성·디버깅이 쉬움 |
| 단일 장애점 | 없음(조정자가 없음) | Orchestrator가 단일 장애점이 될 수 있음 |
| 적합한 규모 | 참여 서비스가 적고 흐름이 단순할 때 | 단계가 많고 복잡한 비즈니스 흐름일 때 |

#### 보상 트랜잭션(Compensating Transaction)
분산 환경에는 여러 DB를 한 번에 롤백할 "공통 ROLLBACK"이 없다. 그래서 Saga는 이미 커밋된 작업의 효과를 의미적으로 되돌리는 별도의 작업, 즉 보상 트랜잭션으로 일관성을 회복한다. "취소"가 아니라 "반대 동작을 새로 한 건 더 실행"하는 것이 핵심이다.

- Ex: 결제 단계의 보상은 "결제 환불", 재고 차감의 보상은 "재고 복구", 주문 생성의 보상은 "주문 취소 상태로 변경"이다. 은행 통장에서 잘못 빠진 돈을 "거래 기록을 지우는 것"이 아니라 "같은 금액을 입금하는 새 거래"로 되돌리는 것과 같다.

#### CQRS(Command Query Responsibility Segregation, 명령·조회 책임 분리)
CQRS는 데이터를 바꾸는 명령(Command, 쓰기)과 데이터를 읽는 조회(Query, 읽기)의 모델을 아예 분리하는 패턴이다. 쓰기는 정규화된 도메인 모델에, 읽기는 화면에 바로 맞춘 조회 전용 모델(읽기 전용 복제본/뷰)에 처리한다. 분산 환경에서 여러 서비스의 데이터를 조합해 보여 줘야 할 때, 매번 여러 서비스에 물어보는 대신 미리 만들어 둔 읽기 모델을 보게 해 성능과 독립성을 얻는다. 대신 쓰기 모델과 읽기 모델 사이에는 최종 일관성(약간의 반영 지연)이 생긴다.

#### 이벤트 소싱(Event Sourcing)
이벤트 소싱은 데이터의 "현재 상태"만 저장하는 대신, 상태를 변화시킨 모든 사건(이벤트)을 발생 순서대로 모두 기록하는 방식이다. 현재 상태는 이벤트들을 처음부터 재생(replay)해 계산한다. 은행 통장의 "현재 잔액"만 들고 있는 대신, "입금 5만/출금 2만/입금 3만"이라는 거래 내역 전체를 적어 두고 합산해 잔액을 구하는 것과 같다. 변경 이력이 완벽히 남아 감사·디버깅·시점 복원에 유리하며, CQRS와 자주 함께 쓰여 "이벤트 → 읽기 모델 갱신" 흐름을 자연스럽게 구성한다.

### 실무 예제
Ex: 주문 → 결제 → 재고 차감 흐름을 Saga(Orchestration 기준)로 처리하고, 재고가 부족할 때 보상 트랜잭션으로 결제 취소·주문 취소까지 되돌리는 흐름이다.

정상 흐름(모든 단계 성공):

```
① 주문 생성   : 주문 서비스 → 주문 DB에 status=PENDING 저장 (로컬 TX)
② 결제 처리   : 결제 서비스 → 결제 DB에 결제 승인 기록 (로컬 TX)
③ 재고 차감   : 재고 서비스 → 재고 DB에서 수량 -1 (로컬 TX)
④ 주문 확정   : 주문 서비스 → 주문 status=CONFIRMED 로 변경
   결과: 세 DB가 각자 커밋되었고, 최종적으로 일관된 상태로 수렴(최종 일관성)
```

보상 흐름(③ 재고 차감 단계에서 재고 부족으로 실패):

```
① 주문 생성   : status=PENDING  ──────────────┐ (성공)
② 결제 처리   : 결제 승인 기록   ──────────┐    │ (성공)
③ 재고 차감   : 재고 부족 → 실패! ✗        │    │
   │                                      │    │
   ▼  Orchestrator가 실패 감지, 보상을 역순으로 실행
   │                                      │    │
③' (재고는 차감 안 됐으니 보상 불필요)        │    │
②' 결제 취소  : 결제 서비스 → 환불 처리 ◀─────┘    │ (②의 보상)
①' 주문 취소  : 주문 서비스 → status=CANCELLED ◀──┘ (①의 보상)
   결과: 차감되었던 효과가 모두 의미적으로 되돌려져 일관성 회복
```

오케스트레이터의 의사 코드(흐름 요약):

```
saga.execute():
    try:
        orderId = 주문서비스.createOrder()        # ①
        payId   = 결제서비스.pay(orderId)          # ②
        재고서비스.decreaseStock(orderId)           # ③  ← 여기서 OutOfStock 예외
        주문서비스.confirm(orderId)                # ④
    except OutOfStock:
        결제서비스.refund(payId)                   # ②' 보상: 결제 취소
        주문서비스.cancel(orderId)                 # ①' 보상: 주문 취소
        # 각 보상 호출은 멱등(idempotent)해야 하며, 실패 시 재시도된다
```

### 주의점 / 안티패턴
- 분산 트랜잭션을 동기 2PC로 억지 구현: MSA에서 "한 번에 롤백되니 편하다"는 이유로 여러 서비스를 2PC로 묶으면, 블로킹·단일 장애점·확장성 저하라는 2PC의 약점을 그대로 떠안는다. 대부분의 경우 Saga + 최종 일관성으로 풀어야 한다.
- 보상 로직 누락: 정상 경로만 구현하고 실패 시 되돌리는 보상 트랜잭션을 빠뜨리면, 결제는 됐는데 주문은 취소되는 식의 "돈은 빠졌는데 물건은 없는" 데이터 불일치가 남는다. 모든 단계에는 짝이 되는 보상 동작이 정의되어 있어야 한다.
- 멱등성(Idempotency) 미보장: 분산 환경에서는 네트워크 재시도·이벤트 중복 전달로 같은 메시지가 두 번 이상 도착할 수 있다. "결제 처리"가 멱등하지 않으면 두 번 결제되고, "재고 차감"이 멱등하지 않으면 두 번 깎인다. 처리 ID(예: orderId)로 중복 요청을 걸러 "여러 번 호출해도 결과는 한 번과 같게" 만들어야 한다.
- 보상의 보상(롤백 실패)을 무시: 보상 트랜잭션 자체도 실패할 수 있다. 보상 실패에 대비한 재시도·데드레터(dead-letter) 큐·수동 개입 알림 등 [회복 탄력성](./resilience.md) 장치를 함께 설계하지 않으면, 일관성이 영구히 깨진 채 방치된다.
- 강한 일관성이 진짜 필요한 영역까지 무리하게 분리: 모든 데이터를 서비스별로 쪼갤 필요는 없다. 한 트랜잭션으로 강하게 묶여야 하는 데이터(예: 같은 계좌의 입출금)는 같은 서비스·같은 DB 안에 두는 것이 옳다. 일관성 경계는 비즈니스 불변식을 기준으로 그어야 한다.
