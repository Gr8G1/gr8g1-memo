## 클라우드 네이티브 (Cloud Native)

### 정의
클라우드 네이티브는 애플리케이션을 **클라우드 환경에서 태어나고 자라도록(native) 설계·구축·운영하는 방식**이다.
단일 기술이 아니라 **컨테이너(Container) + 오케스트레이션(Orchestration) + 마이크로서비스(Microservices) + DevOps/CI-CD** 의 결합으로 정의되며,
클라우드가 제공하는 탄력성(자동 확장·축소)·자가 치유·빠른 배포를 최대한 끌어내는 것을 목표로 한다.

- Ex: 클라우드 네이티브는 **모듈러 가구가 갖춰진 임대 사무실에 입주한 스타트업**과 같다.
  옛날 방식(온프레미스에 직접 지은 단독 건물)은 사람이 늘면 건물을 증축해야 하지만,
  클라우드 네이티브 사무실은 **책상(컨테이너)을 표준 규격으로 찍어내고**, 관리실(오케스트레이터)이 인원이 몰리면 자동으로 책상을 더 들이고, 비면 치우며,
  망가진 책상을 알아서 새것으로 교체한다. 입주사는 가구 배치(인프라)보다 **자기 비즈니스 자체에만 집중**할 수 있다.

### 등장 배경 / 왜 필요한가
클라우드의 핵심 가치는 [온프레미스 vs 클라우드](../on-premise-vs-cloud.md)에서 정리했듯 **탄력성(필요한 만큼 즉시 늘리고 줄이는 능력)** 이다.
그런데 단순히 기존 애플리케이션을 클라우드의 가상 머신(VM)에 그대로 옮겨 놓는다고 이 탄력성을 누릴 수 있는 것은 아니다.

- **무겁고 통째인 애플리케이션은 빠르게 못 늘린다**: 모놀리식 애플리케이션은 시작에 몇 분이 걸리고 상태(state)를 내부에 쌓아두기 때문에, 트래픽이 튀어도 즉시 인스턴스를 늘려 대응하기 어렵다.
- **수동 운영은 클라우드 속도를 못 따라간다**: 서버를 손으로 띄우고 설정하면, 인스턴스를 분 단위로 늘렸다 줄였다 하는 클라우드의 동적 특성을 사람이 감당할 수 없다.
- **환경 차이로 인한 "내 PC에선 됐는데"**: 개발·테스트·운영 환경의 OS·라이브러리 버전이 달라 배포 때마다 사고가 난다.

즉 **클라우드의 탄력성을 제대로 쓰려면 애플리케이션 설계 자체가 바뀌어야 한다**는 깨달음이 클라우드 네이티브를 낳았다.
작게 쪼개고([MSA](./msa.md)), 어디서나 똑같이 도는 단위로 포장하고(컨테이너), 그 단위를 자동으로 늘리고 치유하며(오케스트레이션), 변경을 자동으로 흘려보내는(CI/CD) 흐름이 하나로 묶인 것이다.

### 핵심 개념 / 구성요소
클라우드 네이티브를 떠받치는 네 기둥은 서로 맞물려 동작한다.

- **컨테이너(Container, Docker)**: 애플리케이션 + 실행에 필요한 라이브러리·설정을 하나로 묶어 **어느 환경에서나 동일하게 실행되는 표준 단위**로 만든다. VM보다 가볍고 시작이 빠르다. → [Docker](../../docker/docker.md)
- **오케스트레이션(Orchestration, Kubernetes)**: 수많은 컨테이너를 **자동으로 배치·확장·복구·로드 밸런싱**하는 지휘자. 컨테이너가 죽으면 새로 띄우고(자가 치유), 부하가 오르면 개수를 늘린다(오토스케일링).
- **선언적 인프라(IaC, Infrastructure as Code)**: 인프라 구성을 코드·설정 파일로 기술해 "어떤 상태여야 한다"를 선언하면 시스템이 그 상태로 맞춘다. 버전 관리·재현·자동화가 가능해진다.
- **자동화(DevOps / CI-CD)**: 코드 변경이 자동으로 빌드→테스트→이미지화→배포로 흐르게 만들어, 사람이 손대지 않아도 안전하고 빠르게 릴리스된다. → [배포 전략](../deployment-strategy.md)

#### 컨테이너 → 파드 → 노드 → 클러스터 (Kubernetes 맛보기)
오케스트레이터(Kubernetes)는 컨테이너를 직접 다루지 않고 **파드(Pod)** 라는 단위로 감싸 다룬다. 그 파드들이 **노드(물리/가상 서버)** 위에서 돌고, 노드들이 모여 하나의 **클러스터**를 이룬다.

```
┌──────────────────────────── Cluster (클러스터) ────────────────────────────┐
│                                                                            │
│   ┌─────── Control Plane (관리 두뇌) ───────┐                              │
│   │  스케줄러 / API 서버 / 상태 저장소(etcd)  │  ← "어디에 몇 개 띄울지" 결정   │
│   └─────────────────────────────────────────┘                            │
│                                                                            │
│   ┌──────────── Node 1 ────────────┐   ┌──────────── Node 2 ────────────┐ │
│   │  ┌──────── Pod ────────┐       │   │  ┌──────── Pod ────────┐       │ │
│   │  │ ┌────────────────┐  │       │   │  │ ┌────────────────┐  │       │ │
│   │  │ │ Container (App) │  │       │   │  │ │ Container (App) │  │       │ │
│   │  │ └────────────────┘  │       │   │  │ └────────────────┘  │       │ │
│   │  │ ┌────────────────┐  │       │   │  └─────────────────────┘       │ │
│   │  │ │ Container(로그) │  │       │   │  ┌──────── Pod ────────┐       │ │
│   │  │ └────────────────┘  │       │   │  │ ┌────────────────┐  │       │ │
│   │  └─────────────────────┘       │   │  │ │ Container (App) │  │       │ │
│   └─────────────────────────────────┘   │  └─└────────────────┘──┘       │ │
│                                          └─────────────────────────────────┘ │
│   (부하↑ → 같은 Pod 복제본 자동 증가 / Pod 죽으면 자동 재생성 = 자가 치유)         │
└────────────────────────────────────────────────────────────────────────────┘
```

- **컨테이너(Container)**: 실제 애플리케이션 프로세스가 도는 최소 실행 단위.
- **파드(Pod)**: 하나 이상의 컨테이너를 묶은 Kubernetes의 최소 배포 단위. 같은 파드 안 컨테이너는 네트워크·스토리지를 공유한다.
- **노드(Node)**: 파드가 실제로 실행되는 워커 머신(VM 또는 물리 서버).
- **클러스터(Cluster)**: 여러 노드 + 이를 통제하는 컨트롤 플레인(Control Plane) 전체. "선언한 상태"를 클러스터가 끊임없이 실제 상태와 맞춰 준다(reconciliation).

#### 빌드 → 배포 파이프라인 (CI/CD 흐름)
클라우드 네이티브에서 코드 변경은 사람이 서버에 접속하지 않고 **자동 파이프라인**을 타고 흐른다.

```
 개발자          CI (지속적 통합)                  CD (지속적 배포)
 git push  ─▶  ┌──────────────────┐   image   ┌──────────────┐   apply   ┌──────────────┐
   │           │ 빌드 → 테스트     │ ────────▶ │  Container    │ ────────▶ │ Orchestrator  │
   ▼           │ → 이미지 생성     │  push     │  Registry     │  manifest │ (Kubernetes) │
 [소스코드]     └──────────────────┘           │ (이미지 보관소) │           │ 새 버전 롤아웃 │
                                               └──────────────┘           └──────────────┘
                                                                                  │
                                                                                  ▼
                                                                          무중단 배포(롤링/카나리)
```

#### 12-Factor App
클라우드 네이티브 애플리케이션이 갖춰야 할 설계 원칙을 12가지로 정리한 방법론이다(Heroku 제안). 각 원칙은 **"어디서나 똑같이, 자동으로 늘릴 수 있게"** 라는 목표로 수렴한다.

| # | 원칙 | 한 줄 설명 |
| --- | --- | --- |
| 1 | Codebase (코드베이스) | 하나의 코드베이스를 버전 관리하고, 여러 환경(개발/운영)에 동일하게 배포한다. |
| 2 | Dependencies (의존성) | 의존성을 명시적으로 선언·격리하고, 시스템에 깔린 패키지에 암묵적으로 의존하지 않는다. |
| 3 | Config (설정) | 환경마다 달라지는 설정은 코드가 아닌 **환경 변수**로 분리한다. |
| 4 | Backing Services (지원 서비스) | DB·캐시·큐 같은 외부 서비스를 코드 수정 없이 교체 가능한 **부착 자원**으로 취급한다. |
| 5 | Build, Release, Run (빌드·릴리스·실행) | 빌드·릴리스·실행 단계를 엄격히 분리한다(실행 중 코드 수정 금지). |
| 6 | Processes (프로세스) | 애플리케이션을 **무상태(stateless)** 프로세스로 실행하고, 상태는 외부 저장소에 둔다. |
| 7 | Port Binding (포트 바인딩) | 자체 포트로 서비스를 노출해 독립 실행되며, 외부 웹서버 주입에 의존하지 않는다. |
| 8 | Concurrency (동시성) | 프로세스 모델로 수평 확장한다(인스턴스를 늘려 처리량 증대). |
| 9 | Disposability (폐기 가능성) | 빠르게 시작하고 우아하게 종료(graceful shutdown)해 자유롭게 죽고 살릴 수 있다. |
| 10 | Dev/Prod Parity (개발·운영 일치) | 개발·스테이징·운영 환경의 차이를 최소화한다(컨테이너로 달성). |
| 11 | Logs (로그) | 로그를 파일이 아닌 **이벤트 스트림(stdout)** 으로 흘려보내고, 수집·분석은 외부에 맡긴다. |
| 12 | Admin Processes (관리 프로세스) | DB 마이그레이션 등 일회성 관리 작업을 동일 환경에서 일회성 프로세스로 실행한다. |

### XaaS 심화
IaaS / PaaS / SaaS 의 기본 정의와 비유, AWS·Azure 사례는 [온프레미스 vs 클라우드](../on-premise-vs-cloud.md) 문서에서 이미 다뤘다(중복 서술 생략).
여기서는 클라우드 네이티브에서 비중이 커진 **FaaS** 와 **CaaS** 를 추가로 정리한다.

- **FaaS (Function as a Service, 서버리스)**: 서버 자체를 의식하지 않고 **함수 단위 코드**만 올리면, 요청이 올 때만 실행되고 사용한 실행 시간만큼만 과금되는 모델이다. 인프라·스케일링·서버 관리가 완전히 숨겨진다(AWS Lambda, Azure Functions, Google Cloud Functions). "서버가 없다"가 아니라 **"내가 서버를 신경 쓸 필요가 없다"** 는 뜻이다. 이벤트 기반·간헐적 작업(이미지 변환, 웹훅 처리)에 적합하나, 콜드 스타트(cold start) 지연과 실행 시간 제한이 단점이다.
- **CaaS (Container as a Service)**: 컨테이너를 배포·실행·관리할 수 있는 환경(오케스트레이터 포함)을 서비스로 제공하는 모델이다(AWS ECS/EKS, Azure AKS, Google GKE). IaaS보다 추상화 수준이 높지만(서버를 직접 안 만짐), PaaS만큼 갇혀 있지 않아 **컨테이너 기반 클라우드 네이티브의 표준 실행 토대**가 된다. IaaS와 PaaS 사이의 위치라고 보면 된다.

책임 범위는 모델이 추상화 수준이 높아질수록(아래로 갈수록) **내가 관리할 영역이 줄고 클라우드 제공자가 관리할 영역이 늘어난다.**

```
              관리 주체:  [ 내가 관리 ████ ]   [ 클라우드 제공자 관리 ░░░░ ]

 항목         온프레미스   IaaS        CaaS        PaaS        FaaS        SaaS
 ─────────────────────────────────────────────────────────────────────────────
 함수/코드      ████        ████        ████        ████        ████        ░░░░
 애플리케이션    ████        ████        ████        ████        ░░░░        ░░░░
 런타임         ████        ████        ████        ░░░░        ░░░░        ░░░░
 컨테이너/배포   ████        ████        ░░░░        ░░░░        ░░░░        ░░░░
 OS            ████        ████        ░░░░        ░░░░        ░░░░        ░░░░
 가상화         ████        ░░░░        ░░░░        ░░░░        ░░░░        ░░░░
 서버/스토리지   ████        ░░░░        ░░░░        ░░░░        ░░░░        ░░░░
 네트워킹       ████        ░░░░        ░░░░        ░░░░        ░░░░        ░░░░
 ─────────────────────────────────────────────────────────────────────────────
 한 줄 요약:   다 직접      땅 빌림     실행환경    개발도구    함수만      완제품
                          (서버)      (컨테이너)   까지 빌림    올림        빌림
```

| 모델 | 내가 책임지는 핵심 | 대표 서비스 | 적합한 경우 |
| --- | --- | --- | --- |
| IaaS | OS 위 모든 것(런타임·앱) | AWS EC2, Azure VM | 인프라를 세밀하게 직접 통제하고 싶을 때 |
| CaaS | 컨테이너 이미지와 배포 정의 | AWS EKS/ECS, GKE, AKS | 컨테이너 기반 MSA를 표준으로 운영할 때 |
| PaaS | 애플리케이션 코드 | Heroku, App Engine | 인프라 신경 끄고 앱 개발에만 집중할 때 |
| FaaS | 함수 코드 한 조각 | AWS Lambda, Azure Functions | 이벤트성·간헐적 작업, 사용량 기반 과금 |
| SaaS | 사용(설정)만 | Gmail, Slack | 완성된 소프트웨어를 그대로 쓸 때 |

### 배치 전략
클라우드 네이티브 애플리케이션을 **어디에 둘 것인가**는 [MSA](./msa.md) 운영 관점(독립 배포·확장·장애 격리)에서 트레이드오프가 갈린다.

| 구분 | 온프레미스(자체 K8s) | 퍼블릭 클라우드 | 하이브리드 / 멀티클라우드 |
| --- | --- | --- | --- |
| 인프라 소유 | 직접 보유·운영 | 클라우드 제공자 | 일부 자체 + 일부 클라우드 |
| 탄력성(오토스케일) | 물리 한계에 묶임 | 사실상 무제한·즉시 | 클라우드 쪽만 자유로움 |
| MSA 운영 부담 | K8s까지 직접 구축·운영(높음) | 관리형 K8s(EKS/GKE/AKS)로 경감 | 환경별 배포·네트워크 이중 관리(가장 복잡) |
| 데이터 주권/규제 | 가장 유리(데이터 내부 보관) | 리전·규제 의존 | 민감 데이터는 온프레미스, 나머지는 클라우드 |
| 비용 구조 | 초기 대규모 투자(CapEx) | 사용량 과금(OpEx) | 혼합 |
| 벤더 종속 | 없음 | 높음(특정 클라우드에 묶임) | 낮춤(여러 클라우드로 분산) |

- **온프레미스 + 자체 Kubernetes**: 규제·데이터 주권이 중요하거나 이미 데이터센터에 투자한 경우. 단, MSA의 디스커버리·게이트웨이·모니터링까지 전부 직접 운영해야 해 부담이 크다.
- **퍼블릭 클라우드 + 관리형 Kubernetes**: 대다수 클라우드 네이티브의 기본 선택. 탄력적 확장과 관리형 서비스로 MSA 운영 부담을 크게 줄인다.
- **하이브리드 / 멀티클라우드**: 민감 데이터는 온프레미스에 두고 변동 트래픽은 퍼블릭 클라우드로 흘리거나, 특정 벤더 종속을 피하려 여러 클라우드에 분산한다. 유연하지만 **네트워크·관측성·배포를 환경마다 이중으로 다뤄야 해 운영 복잡도가 가장 높다.**

### 실무 예제
Ex: 컨테이너 이미지를 빌드해 레지스트리에 푸시하고, 오케스트레이터(Kubernetes)에 배포하기까지의 흐름을 따라가 본다.
`소스코드 → Dockerfile로 이미지 빌드 → 레지스트리 푸시 → 매니페스트로 클러스터 배포` 순서다.

1단계 — 애플리케이션을 컨테이너 이미지로 굳히는 `Dockerfile` (빌드 단계를 분리한 멀티스테이지 예시).

```dockerfile
# 1) 빌드 단계: 소스를 컴파일해 실행 가능한 jar 생성
FROM eclipse-temurin:21-jdk AS build
WORKDIR /app
COPY . .
RUN ./gradlew bootJar --no-daemon          # 빌드 산출물 생성

# 2) 실행 단계: 가벼운 런타임 이미지에 jar만 복사(이미지 경량화)
FROM eclipse-temurin:21-jre
WORKDIR /app
COPY --from=build /app/build/libs/*.jar app.jar
EXPOSE 8080                                  # 12-Factor: Port Binding
ENTRYPOINT ["java", "-jar", "app.jar"]
```

2단계 — 이미지를 빌드하고 컨테이너 레지스트리에 푸시한다.

```bash
# 이미지 빌드 (태그에 버전을 명시해 추적 가능하게)
docker build -t myregistry.io/order-service:1.2.0 .

# 레지스트리 로그인 후 푸시 (Docker Hub, AWS ECR, GHCR 등)
docker push myregistry.io/order-service:1.2.0
```

3단계 — Kubernetes 매니페스트로 클러스터에 선언적으로 배포한다(IaC). `replicas: 3`은 같은 파드를 3개 띄워 부하를 분산한다는 선언이다.

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: order-service
spec:
  replicas: 3                       # 파드 복제본 3개(수평 확장 / 12-Factor: Concurrency)
  selector:
    matchLabels:
      app: order-service
  template:
    metadata:
      labels:
        app: order-service
    spec:
      containers:
        - name: order-service
          image: myregistry.io/order-service:1.2.0   # 위에서 푸시한 이미지
          ports:
            - containerPort: 8080
          env:
            - name: SPRING_PROFILES_ACTIVE
              value: "prod"          # 12-Factor: 설정은 환경 변수로 주입
          readinessProbe:            # 준비됐을 때만 트래픽을 받게(무중단 배포 기반)
            httpGet:
              path: /actuator/health
              port: 8080
            initialDelaySeconds: 10
---
apiVersion: v1
kind: Service
metadata:
  name: order-service
spec:
  selector:
    app: order-service
  ports:
    - port: 80
      targetPort: 8080              # 3개 파드 앞단에서 로드 밸런싱하는 안정적 진입점
```

```bash
# 선언한 상태를 클러스터에 적용 → 오케스트레이터가 실제 상태를 맞춤
kubectl apply -f deployment.yaml
```

이후 이미지 태그만 새 버전으로 바꿔 다시 `apply` 하면, 오케스트레이터가 기존 파드를 하나씩 교체하는 **롤링 업데이트(무중단 배포)** 를 자동으로 수행한다. 자세한 배포 방식은 [배포 전략](../deployment-strategy.md)을 참고한다.

### 주의점 / 안티패턴
클라우드 네이티브는 "클라우드에 올렸다"가 아니라 **"클라우드의 탄력성을 설계로 끌어냈다"** 일 때 성립한다.

- **리프트 앤 시프트(Lift and Shift)를 클라우드 네이티브라 착각**: 기존 모놀리식을 손대지 않고 VM째 클라우드로 옮긴 것은 단순 이전일 뿐이다. 무상태화·컨테이너화·자동 확장이 없으면 클라우드 비용만 더 쓰고 탄력성은 못 누린다. **"클라우드에서 도는 것"과 "클라우드 네이티브"는 다르다.**
- **벤더 종속(Vendor Lock-in)**: 특정 클라우드의 독점 서비스(전용 FaaS·매니지드 DB·메시징)에 깊이 묶이면 나중에 이전 비용이 폭발한다. 표준(Kubernetes·OCI 컨테이너·오픈소스)을 중심에 두고, 종속이 불가피한 부분은 추상화 계층으로 격리해 영향 범위를 좁힌다.
- **오버 엔지니어링(Over-engineering)**: 트래픽이 적고 팀이 작은데 처음부터 Kubernetes·서비스 메시·풀 MSA를 도입하면, 비즈니스 가치보다 인프라 운영에 더 많은 인력이 묶인다. [MSA](./msa.md)와 마찬가지로 **모놀리식·단순 컨테이너로 시작해 필요할 때 점진적으로 확장**하는 편이 안전하다.
- **상태를 컨테이너 안에 쌓기**: 컨테이너는 언제든 죽고 다시 뜨는 일회용이다. 세션·파일을 컨테이너 내부에 저장하면 파드 재생성 시 모두 사라진다. 상태는 반드시 외부 저장소(DB·오브젝트 스토리지)에 둔다(12-Factor: Processes).
- **관측성(Observability) 없이 운영**: 파드가 자동으로 뜨고 사라지는 환경에서 중앙 로그·메트릭·분산 추적이 없으면 장애 원인 추적이 불가능하다. → [안정성](./resilience.md)
