## API Gateway

### 정의

API Gateway는 모든 클라이언트 요청이 가장 먼저 거치는 단일 진입점(Single Entry Point)이다.
클라이언트(웹, 앱, 외부 시스템)는 내부에 어떤 서비스가 몇 개나 있는지 알 필요 없이, 게이트웨이라는 하나의 주소로만 요청을 보낸다. 게이트웨이는 그 요청을 받아 알맞은 내부 서비스로 길을 안내(라우팅)하고, 인증·요청 제한·로깅 같은 공통 처리를 대신 수행한 뒤 응답을 돌려준다.

- Ex: 큰 회사 건물의 "정문 안내 데스크"와 같다. 방문객(클라이언트)은 각 부서(서비스)의 사무실 위치를 외울 필요 없이 1층 안내 데스크에 용건만 말한다. 데스크 직원은 방문증을 확인(인증)하고, 너무 많은 사람이 한 번에 몰리면 줄을 세우며(요청 제한), 방문 기록을 남기고(로깅), 해당 부서로 안내(라우팅)한다. 방문객은 건물 내부 구조를 전혀 몰라도 된다.

### 등장 배경 / 왜 필요한가

서비스를 잘게 나누는 [MSA](./msa.md) 구조에서는 주문, 결제, 회원, 상품 등 기능별로 서비스가 수십 개로 늘어난다. 이때 클라이언트가 각 서비스를 **직접** 호출하면 다음과 같은 문제가 생긴다.

- 엔드포인트 폭증: 클라이언트가 서비스 주소를 전부 알고 있어야 한다. 서비스가 추가·삭제되거나 주소가 바뀌면 클라이언트 코드도 매번 따라 바뀌어야 한다.
- 인증·인가 중복: 모든 서비스가 각자 토큰 검증, 권한 확인 로직을 똑같이 구현해야 한다. 공통 로직이 N개로 흩어져 유지보수가 어렵다.
- CORS·보안 정책 분산: 브라우저의 교차 출처(CORS) 설정, TLS 종료, IP 차단 같은 정책을 서비스마다 따로 관리해야 한다.
- 클라이언트 부담: 화면 하나를 그리려고 여러 서비스를 각각 호출하면 요청 수가 늘고, 모바일처럼 네트워크가 느린 환경에서 성능이 떨어진다.

이 공통 관심사(횡단 관심사, cross-cutting concern)들을 클라이언트와 서비스 사이의 한 지점에 모아 일원화한 것이 API Gateway다. 클라이언트는 게이트웨이 하나만 바라보고, 내부 서비스 구성은 게이트웨이 뒤에 감춰진다.

### 핵심 개념 / 구성요소

게이트웨이는 단순 전달자가 아니라 여러 공통 책임을 함께 처리한다.

- 라우팅(Routing): 요청 경로·헤더·메서드를 보고 알맞은 내부 서비스로 전달한다.
- 인증·인가(Auth): 토큰(JWT 등)을 검증하고 권한을 확인해, 통과한 요청만 내부로 보낸다.
- 요청 제한(Rate Limiting): 일정 시간당 허용 요청 수를 제한해 과도한 트래픽·공격으로부터 내부 서비스를 보호한다.
- 로깅·모니터링(Logging/Monitoring): 모든 요청이 한 곳을 거치므로 트래픽·지연·에러를 중앙에서 수집하기 쉽다.
- 응답 집계(Aggregation): 여러 서비스 호출 결과를 하나의 응답으로 합쳐 클라이언트의 호출 횟수를 줄인다.
- 프로토콜 변환(Protocol Translation): 외부는 HTTP/REST로 받고, 내부는 gRPC 등 다른 프로토콜로 변환해 전달할 수 있다.

```
                                          +-----------------+
                                          | 회원 서비스        |
                                          +-----------------+
                                                  ^
   웹 클라이언트  --+                                |
   모바일 앱      --+                       +------------------+    +------------------+
   외부 시스템    --+----- HTTP/REST --->   |   API Gateway    |--->| 주문 서비스      |
                                         | - 라우팅           |    +------------------+
                                         | - 인증/인가        |
                                         | - Rate Limiting  |    +------------------+
                                         | - 로깅/모니터링     |--->| 결제 서비스      |
                                         | - 응답 집계        |    +------------------+
                                         | - 프로토콜 변환     |
                                         +------------------+    +------------------+
                                                  |          --->| 상품 서비스      |
                                                  |              +------------------+
                                                  v
                                       (단일 진입점 / Single Entry Point)
```

#### 리버스 프록시와의 관계

API Gateway는 [서버](../server.md)에서 다룬 **리버스 프록시(Reverse Proxy)** 의 발전된 형태로 볼 수 있다.

- 리버스 프록시: 서버를 대신해 클라이언트와 소통하며, 들어온 요청을 여러 서버로 분산(로드 밸런싱)하고 IP·내부 구조를 숨겨 보안을 강화한다. 즉 "어디로 보낼지"와 "부하 분산"이 핵심이다.
- API Gateway: 리버스 프록시의 라우팅·로드 밸런싱 능력을 그대로 갖되, 그 위에 인증·요청 제한·응답 집계·프로토콜 변환 같은 **애플리케이션 레벨(L7)** 의 공통 책임을 추가로 얹은 것이다.

정리하면, 리버스 프록시가 "트래픽을 알맞은 서버로 흘려보내는 길목"이라면, API Gateway는 그 길목에서 "통행증 검사·통행량 제한·여러 부서 결과 취합"까지 수행하는 똑똑한 관문이다. 실제로 게이트웨이 뒤에 동일 서비스의 인스턴스가 여러 개 있을 때 [서버](../server.md)의 로드 밸런싱 알고리즘(라운드로빈, 최소 연결 등)이 그대로 활용된다.

#### API Gateway vs BFF 경계

둘 다 클라이언트와 내부 서비스 사이에 위치하지만 책임이 다르다.

- API Gateway: 모든 클라이언트에 **공통**으로 적용되는 횡단 관심사(인증, 요청 제한, 로깅, 라우팅)를 한 곳에서 처리한다. "누구에게나 똑같이 적용되는 관문"이다.
- BFF(Backend For Frontend): **특정 클라이언트 유형**(웹용 / 모바일용 등)에 맞춰 여러 서비스 응답을 조합·가공하는 전용 백엔드다. "클라이언트별 맞춤 주방"에 가깝다.

실무에서는 클라이언트 → API Gateway(공통 처리) → BFF(클라이언트별 조합) → 내부 서비스 순으로 함께 쓰기도 한다. 상세한 패턴은 [BFF](./bff.md) 문서로 연결한다.

### 장점 / 단점(트레이드오프)

- 장점
  - 공통 관심사 일원화: 인증·요청 제한·로깅을 한 곳에서 관리해 중복 코드가 사라진다.
  - 클라이언트 단순화: 클라이언트는 게이트웨이 주소 하나만 알면 되고, 내부 서비스 변경에 영향을 덜 받는다.
  - 내부 구조 은닉: 서비스 개수·위치·프로토콜이 외부에 드러나지 않아 보안에 유리하다.
  - 호출 횟수 감소: 응답 집계로 클라이언트의 네트워크 왕복(round trip)을 줄인다.
- 단점(트레이드오프)
  - 단일 장애점(SPOF, Single Point of Failure) 위험: 게이트웨이가 멈추면 모든 요청이 막힌다. 다중화·헬스 체크가 필수다.
  - 성능 병목 가능성: 모든 트래픽이 한 곳을 거치므로 게이트웨이 자체가 병목이 될 수 있다.
  - 운영 복잡도 증가: 게이트웨이라는 계층이 하나 더 생기므로 관리·배포·모니터링 대상이 늘어난다.
  - 결합도 주의: 게이트웨이에 너무 많은 로직을 넣으면 오히려 변경이 어려워진다(아래 안티패턴 참고).

### 실무 예제

Ex: Spring Cloud Gateway에서 `application.yml`로 라우팅을 선언적으로 설정한다. `predicates`(어떤 요청을 이 경로로 받을지 조건)와 `filters`(전달 전후로 적용할 처리)를 조합한다.

```yaml
spring:
  cloud:
    gateway:
      routes:
        # 주문 서비스 라우팅
        - id: order-service
          uri: lb://ORDER-SERVICE          # lb:// 는 서비스 디스커버리에 등록된 이름으로 로드 밸런싱
          predicates:
            - Path=/api/orders/**          # /api/orders 로 시작하는 요청만 이 경로로
            - Method=GET,POST
          filters:
            - StripPrefix=1                # /api 접두어 제거 후 내부 서비스로 전달
            - name: RequestRateLimiter     # Rate Limiting(요청 제한) 필터
              args:
                redis-rate-limiter.replenishRate: 10   # 초당 보충 토큰 수
                redis-rate-limiter.burstCapacity: 20    # 순간 최대 허용 수

        # 결제 서비스 라우팅
        - id: payment-service
          uri: lb://PAYMENT-SERVICE
          predicates:
            - Path=/api/payments/**
          filters:
            - StripPrefix=1
            - AddRequestHeader=X-Gateway, spring-cloud-gateway   # 공통 헤더 추가

      # 모든 경로에 공통 적용되는 기본 필터
      default-filters:
        - name: RemoveRequestHeader
          args:
            name: Cookie                   # 내부로 쿠키를 넘기지 않음(보안)
```

위 설정에서 `uri: lb://ORDER-SERVICE`의 `lb://`는 서비스 이름으로 실제 인스턴스를 찾아 로드 밸런싱하겠다는 뜻이며, 실제 주소 해석은 [서비스 디스커버리](./service-discovery.md)와 연동된다.

### 주의점 / 안티패턴

- 비즈니스 로직 과집중 금지: 게이트웨이는 라우팅·인증·요청 제한 같은 **공통 횡단 관심사**만 다뤄야 한다. 할인 계산, 재고 차감 같은 도메인 로직을 게이트웨이에 넣으면 "갓 게이트웨이(God Gateway)"가 되어, 특정 서비스가 바뀔 때마다 게이트웨이도 배포해야 하는 강결합이 생긴다.
- 단일 장애점(SPOF) 방치 금지: 게이트웨이를 1대만 두면 멈추는 순간 전체 서비스가 마비된다. 반드시 다중 인스턴스 + 앞단 로드 밸런서 + 헬스 체크로 이중화한다.
- 병목 무대책 금지: 모든 트래픽이 모이는 지점이므로 타임아웃, 커넥션 풀, 회로 차단(circuit breaker) 등 [회복 탄력성](./resilience.md) 장치 없이 운영하면 한 서비스의 장애가 게이트웨이를 통해 전체로 번질 수 있다.
- 과도한 응답 집계 주의: 게이트웨이에서 너무 많은 서비스를 모아 응답을 조합하면, 클라이언트별 요구가 게이트웨이를 복잡하게 만든다. 클라이언트 맞춤 조합이 필요하면 [BFF](./bff.md)로 분리하는 편이 낫다.
