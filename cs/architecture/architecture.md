## 소프트웨어 아키텍처

### 정의
소프트웨어 아키텍처(Software Architecture)란 시스템을 구성하는 요소들을 어떻게 나누고, 각 요소가 어떤 책임을 지며, 서로 어떻게 소통하는지를 정의한 "큰 그림(설계 청사진)"이다.
코드 한 줄 한 줄의 문법이 아니라, 건물로 치면 "방을 몇 개로 나누고, 복도를 어떻게 연결하며, 어디에 출입문과 비상구를 둘 것인가"에 해당하는 구조적 결정이다.

- Ex: 1인 가구를 위한 원룸(Monolith)은 부엌·침실·거실이 한 공간에 다 들어 있어 짓기도 쉽고 동선도 단순하다. 하지만 대가족이 들어와 각자 다른 생활 패턴을 갖게 되면, 방을 분리하고 층을 나눈 다세대 주택(MSA)이 더 살기 편해진다. 어느 쪽이 "정답"인 게 아니라 사는 사람(조직)과 규모에 맞는 구조를 고르는 일이다.

아키텍처 패턴을 굳이 논하는 이유는, 잘못된 구조는 나중에 바꾸는 비용이 코드 버그를 고치는 비용보다 훨씬 크기 때문이다. 함수 하나는 5분이면 고치지만, 모든 기능이 한 덩어리로 엉켜 있는 시스템을 쪼개는 일은 몇 달이 걸린다. 그래서 "어떤 구조로 시작할 것인가"는 프로젝트 초기에 가장 신중하게 정해야 할 의사결정이다.

### 등장 배경 / 왜 필요한가
초창기 서비스는 대부분 하나의 실행 단위(하나의 배포 파일)에 모든 기능을 담는 모놀리식(Monolith) 구조로 출발했다. 규모가 작을 때는 이 방식이 가장 빠르고 단순했다.

그러나 서비스가 성장하면서 다음과 같은 고통이 누적되었다.

- 코드베이스가 거대해져 한 명의 개발자가 전체를 파악하기 어려워진다.
- 작은 기능 하나를 고쳐도 전체를 다시 빌드·배포해야 하고, 한 번의 배포 실패가 서비스 전체를 멈춘다.
- 특정 기능(예: 결제)에만 트래픽이 몰려도 전체 시스템을 통째로 늘려야 해서 자원이 낭비된다.
- 팀이 커질수록 같은 코드를 여러 팀이 동시에 건드려 충돌이 잦아진다.

이러한 "성장통"을 해소하기 위해 시스템을 더 작은 단위로 쪼개는 방향으로 아키텍처가 진화해 왔다. 즉, 아키텍처 패턴은 추상적인 이론이 아니라 규모 확장 과정에서 실제로 겪은 문제를 풀기 위한 실전 처방전이다.

### 핵심 개념 / 구성요소

#### 아키텍처 진화 흐름
아키텍처는 "더 작게 쪼개고, 더 독립적으로 만든다"는 방향으로 진화해 왔다. 아래는 그 흐름을 시간 축으로 표현한 타임라인이다.

```
 [Monolith] ──> [Modular Monolith] ──> [SOA] ──> [MSA]
     │                  │                 │          │
  하나의 덩어리      하나의 덩어리지만   서비스 단위로   완전히 독립된
  (전부 한 곳)       내부를 모듈로 분리   기능을 분리     작은 서비스들
                                       (공유 버스 사용) (각자 배포/DB)
   ◀── 단순함 / 빠른 시작 ──────────── 복잡함 / 높은 확장성 ──▶
```

- Monolith(모놀리식): 모든 기능이 하나의 배포 단위에 들어 있다. 빠르게 시작할 수 있지만 커질수록 무거워진다.
- Modular Monolith(모듈러 모놀리식): 여전히 하나로 배포되지만, 내부를 도메인별 모듈로 명확히 경계 지어 둔다. 모놀리식의 단순함을 유지하면서 추후 분리를 쉽게 만드는 절충안이다.
- SOA(Service-Oriented Architecture, 서비스 지향 아키텍처): 기능을 서비스 단위로 나누되, 보통 ESB(Enterprise Service Bus)라는 중앙 통신 버스를 공유한다. 중앙 집중 버스가 병목이 되기 쉽다는 한계가 있었다.
- MSA(Microservices Architecture, 마이크로서비스 아키텍처): 서비스를 작고 독립적인 단위로 쪼개고, 각 서비스가 자기 데이터베이스와 배포 파이프라인을 갖는다. 가장 높은 확장성과 독립성을 주지만 운영 복잡도가 크게 올라간다.

#### Monolith vs MSA 비교
한 덩어리(Monolith)와 여러 조각(MSA)이라는 양 극단을 관점별로 비교하면 다음과 같다.

| 관점 | Monolith(모놀리식) | MSA(마이크로서비스) |
| --- | --- | --- |
| 배포(Deploy) | 전체를 한 번에 배포. 작은 수정도 전체 재배포 | 서비스별 독립 배포. 바꾼 부분만 배포 가능 |
| 확장(Scaling) | 전체를 통째로 복제(수직/전체 수평 확장) | 부하가 큰 서비스만 골라서 확장 |
| 장애 격리 | 한 곳의 장애가 전체로 전파되기 쉬움 | 한 서비스 장애가 다른 서비스로 전파되지 않도록 격리 가능 |
| 복잡도 | 초기 구조 단순, 규모 커지면 내부 복잡 | 초기 구조 복잡(네트워크/분산 처리), 서비스 단위는 단순 |
| 팀 구조 | 하나의 큰 코드베이스를 여러 팀이 공유 | 팀별로 서비스를 소유(독립적으로 개발) |
| 트랜잭션 | 단일 DB 트랜잭션으로 강한 일관성 보장 쉬움 | 서비스마다 DB가 달라 분산 트랜잭션/최종 일관성 고민 필요 |

#### 콘웨이의 법칙(Conway's Law)
콘웨이의 법칙은 "시스템을 설계하는 조직은 그 조직의 의사소통 구조를 그대로 닮은 시스템을 만들어 낸다"는 경험칙이다. 쉽게 말해, 팀이 3개면 시스템도 자연스럽게 3덩어리로 나뉘는 경향이 있다.

```
 [조직 구조]                        [시스템 구조]
  주문팀  ────────(닮는다)────────>  주문 서비스
  결제팀  ────────(닮는다)────────>  결제 서비스
  배송팀  ────────(닮는다)────────>  배송 서비스
```

이 법칙이 MSA와 직접 맞닿는 지점은, "서비스를 잘 쪼개려면 조직(팀)도 그 경계에 맞춰 나뉘어 있어야 한다"는 점이다. 팀은 하나인데 서비스만 여러 개로 쪼개면, 한 팀이 여러 서비스를 오가며 결국 분산된 모놀리식(Distributed Monolith)이 되어 버린다. 그래서 MSA를 도입할 때는 코드보다 먼저 조직을 그 경계에 맞춰 정렬하는 것이 핵심이며, 이 발상을 적극 활용한 것이 "역(逆) 콘웨이 전략(Inverse Conway Maneuver)" — 원하는 시스템 구조를 먼저 그리고 거기에 맞게 팀을 구성하는 방법이다.

### 장점 / 단점(트레이드오프)
아키텍처 선택은 "좋은 것 vs 나쁜 것"이 아니라 "무엇을 얻고 무엇을 포기할 것인가"의 트레이드오프다.

- Monolith(모놀리식)
  - 장점: 개발·테스트·배포가 단순하고 빠르다. 하나의 DB로 트랜잭션 일관성을 지키기 쉽고, 함수 호출이라 네트워크 지연이 없다. 초기 비용과 운영 부담이 작다.
  - 단점: 코드가 커지면 빌드·배포가 느려지고, 부분 확장이 어렵다. 한 곳의 장애가 전체로 번질 수 있고, 여러 팀이 한 코드베이스를 공유해 충돌이 잦다.
- MSA(마이크로서비스)
  - 장점: 서비스별 독립 배포·확장이 가능하고, 장애를 격리할 수 있다. 팀이 자기 서비스를 자율적으로 소유해 병렬 개발 속도가 빠르며, 서비스별로 다른 기술 스택을 쓸 수 있다.
  - 단점: 네트워크 통신·분산 트랜잭션·서비스 디스커버리·모니터링 등 운영 복잡도가 급격히 올라간다. 데이터 일관성을 직접 설계해야 하고, 분산 환경 디버깅이 어렵다. 인프라·운영 성숙도가 낮으면 오히려 독이 된다.

### 실무 예제
Ex: 작은 스타트업이 쇼핑몰 서비스를 모놀리식으로 시작했다. 처음에는 회원·상품·주문·결제가 한 프로젝트 안의 폴더로만 나뉘어 있었다.

```
[초기 - Monolith]
  shop-app (단일 배포)
   ├─ member/   (회원)
   ├─ product/  (상품)
   ├─ order/    (주문)
   └─ payment/  (결제)
        │
        └─ 하나의 DB
```

Ex: 트래픽이 폭발하면서 결제 부분에만 부하가 몰리고, 주문 코드를 고칠 때마다 전체를 재배포하느라 결제까지 멈추는 사고가 반복됐다. 그래서 도메인 경계를 따라 서비스를 분리하고 각자 DB와 배포 파이프라인을 갖게 했다.

```
[성장 후 - MSA]
            ┌─ 회원 서비스 ── 회원 DB
  API       ├─ 상품 서비스 ── 상품 DB
 Gateway ──▶├─ 주문 서비스 ── 주문 DB
            └─ 결제 서비스 ── 결제 DB
   (각 서비스는 독립 배포 / 독립 확장)
```

이렇게 분리하면 결제 서비스만 인스턴스를 늘려(Scale-Out) 부하를 흡수할 수 있고, 주문 서비스를 배포해도 결제는 영향을 받지 않는다. 다만 "주문 시 결제까지 한 트랜잭션으로 처리"하던 로직은 더 이상 단일 DB 트랜잭션이 아니므로, 서비스 간 데이터 일관성을 별도로 설계해야 한다.

### 주의점 / 안티패턴
- 처음부터 MSA로 시작하지 마라: 도메인 경계가 아직 불분명한 초기 단계에 서비스를 쪼개면, 경계를 잘못 그어 두고두고 고생한다. 대부분의 경우 모놀리식(또는 모듈러 모놀리식)으로 시작해 필요해질 때 쪼개는 편이 안전하다.
- 분산된 모놀리식(Distributed Monolith) 주의: 서비스만 나눴을 뿐 서로 강하게 결합되어, 한 서비스를 바꾸면 다른 서비스도 같이 배포해야 한다면 MSA의 장점은 사라지고 운영 복잡도만 남는다.
- 너무 잘게 쪼개기(Nano-service): 서비스가 지나치게 작으면 네트워크 호출만 많아져 성능과 디버깅이 모두 나빠진다. "독립적으로 배포·확장할 가치가 있는 단위"인지 기준으로 삼아야 한다.
- 조직 정렬 없이 도입: 콘웨이의 법칙을 무시하고 팀 구조를 그대로 둔 채 서비스만 쪼개면, 결국 한 팀이 여러 서비스를 떠안아 자율성이라는 핵심 이점을 잃는다.
- 운영 성숙도 무시: CI/CD, 모니터링, 로그 수집, 장애 대응 체계 없이 MSA를 도입하면 장애 추적조차 어려운 블랙박스가 된다.

#### 선택 기준 — 언제 Monolith, 언제 MSA?
"유행이라서"가 아니라 아래 기준으로 판단한다.

| 판단 기준 | Monolith가 적합 | MSA가 적합 |
| --- | --- | --- |
| 조직 규모 | 소수 인원, 단일 팀 | 다수의 독립 팀 |
| 트래픽 | 예측 가능하고 균일 | 폭발적/특정 영역 편중 |
| 도메인 복잡도 | 단순, 경계 불명확 | 복잡, 경계 뚜렷 |
| 운영 성숙도 | CI/CD·모니터링 미비 | 자동화·관측 체계 충분 |

정리하면, 초기·소규모·불확실한 단계에서는 모놀리식이 거의 항상 옳고, 조직과 트래픽과 도메인이 함께 커지고 운영 역량까지 갖춰졌을 때 비로소 MSA로의 전환을 고려한다.

### 관련 문서
이 폴더(architecture)의 나머지 문서들을 주제별로 정리한 인덱스다.

- [마이크로서비스 아키텍처(MSA)](./msa.md) — 시스템을 작고 독립적인 서비스로 쪼개는 핵심 패턴과 분리 기준.
- [BFF(Backend For Frontend)](./bff.md) — 클라이언트(웹/모바일)별로 전용 백엔드를 두어 응답을 최적화하는 패턴.
- [API 게이트웨이(API Gateway)](./api-gateway.md) — 여러 서비스 앞단에서 라우팅·인증·집계를 담당하는 단일 진입점.
- [서비스 디스커버리(Service Discovery)](./service-discovery.md) — 동적으로 늘고 줄어드는 서비스의 위치를 찾아 연결하는 메커니즘.
- [회복 탄력성(Resilience)](./resilience.md) — 서킷 브레이커·재시도·타임아웃 등 장애 격리와 복원 전략.
- [데이터 일관성(Data Consistency)](./data-consistency.md) — 서비스마다 DB가 분리된 환경에서의 분산 트랜잭션·최종 일관성·Saga 패턴.
- [이벤트 기반 아키텍처(Event-Driven)](./event-driven.md) — 메시지·이벤트로 서비스 간 결합을 느슨하게 만드는 비동기 통신 방식.
- [클라우드 네이티브(Cloud Native)](./cloud-native.md) — 컨테이너·오케스트레이션을 전제로 클라우드 환경에 최적화해 설계·운영하는 접근.
