## MSA (Microservices Architecture, 마이크로서비스 아키텍처)

### 정의
MSA는 하나의 큰 애플리케이션을 **비즈니스 기능 단위로 잘게 쪼갠 작은 서비스(마이크로서비스)들의 모음**으로 구성하는 아키텍처 스타일이다.
각 서비스는 독립적으로 개발·배포·확장되며, 자신만의 데이터를 소유하고, 가벼운 통신(HTTP/메시징)으로 서로 협력한다.

- Ex: 모놀리식(Monolithic)이 **모든 메뉴를 한 명의 만능 요리사가 혼자 처리하는 단일 주방**이라면,
  MSA는 **전채·메인·디저트·음료 코너를 각기 다른 전문 셰프가 독립된 부스에서 맡고, 주문서(API)로만 소통하는 푸드코트**와 같다.
  음료 코너가 바빠지면 음료 셰프만 더 투입하면 되고(독립 확장), 디저트 부스 공사가 메인 코너 영업을 막지 않는다(독립 배포).

### 등장 배경 / 왜 필요한가
초기에는 하나의 코드베이스에 모든 기능을 담는 **모놀리식 아키텍처(Monolithic Architecture)** 가 일반적이었다.
규모가 작을 때는 단순하고 빠르지만, 서비스가 커지면 다음 한계가 드러난다.

- **배포 단위가 거대하다**: 결제 로직 한 줄을 고쳐도 애플리케이션 전체를 다시 빌드·배포해야 한다. 배포 주기가 느려지고 위험이 커진다.
- **부분 확장이 불가능하다**: 주문 트래픽만 폭증해도 전체 인스턴스를 통째로 스케일아웃해야 해서 자원이 낭비된다.
- **기술 스택이 고정된다**: 전체가 한 언어·프레임워크에 묶여, 특정 기능에 더 적합한 기술을 골라 쓰기 어렵다.
- **장애가 전체로 번진다**: 한 모듈의 메모리 누수가 프로세스 전체를 죽여 모든 기능이 동시에 멈춘다.
- **코드베이스가 커질수록 인지 부하가 폭발한다**: 팀이 커지면 서로의 코드를 건드리며 충돌이 잦아진다.

MSA는 이 한계를 **"작게 나누고, 독립적으로 운영한다"** 는 원칙으로 해결하려는 시도이며,
[클라우드 네이티브](./cloud-native.md) 환경과 컨테이너·오케스트레이션 기술의 성숙이 이를 현실로 만들었다.

### 핵심 개념 / 구성요소
MSA는 단순히 서비스를 쪼개는 것에 그치지 않고, 분산된 서비스들을 묶어주는 **공통 인프라 구성요소**를 함께 갖춘다.

```
                          ┌───────────────────────────┐
        클라이언트(웹/앱)  ─▶│      API Gateway          │  진입점·라우팅·인증·집계
                          └─────────────┬─────────────┘
                                        │ (라우팅)
              ┌──────────────┬──────────┼──────────────┬──────────────┐
              ▼              ▼          ▼              ▼              ▼
         ┌─────────┐    ┌─────────┐ ┌─────────┐   ┌─────────┐   ┌─────────┐
         │ 회원     │    │ 주문     │ │ 결제     │   │ 상품     │   │ 배송     │
         │ Service │    │ Service │ │ Service │   │ Service │   │ Service │
         └────┬────┘    └────┬────┘ └────┬────┘   └────┬────┘   └────┬────┘
              │ DB           │ DB        │ DB          │ DB          │ DB
            ┌─┴─┐          ┌─┴─┐       ┌─┴─┐         ┌─┴─┐         ┌─┴─┐
            │DB1│          │DB2│       │DB3│         │DB4│         │DB5│   ← Database per Service
            └───┘          └───┘       └───┘         └───┘         └───┘
              │              │           │             │             │
              └──────────────┴─────┬─────┴─────────────┴─────────────┘
                                   ▼ (비동기 이벤트 발행/구독)
                          ┌───────────────────────────┐
                          │   Message Broker (Kafka 등) │
                          └───────────────────────────┘

        [ 횡단 인프라 ]
        ┌──────────────────┐   ┌──────────────────┐   ┌──────────────────┐
        │ Service Discovery │   │   Config Server   │   │  분산 추적/모니터링 │
        │  (어디 있나 등록·조회)│   │  (설정 중앙 관리)   │   │  (로그·메트릭·트레이스)│
        └──────────────────┘   └──────────────────┘   └──────────────────┘
```

- **개별 서비스(Microservice)**: 하나의 비즈니스 능력을 담당하는 독립 배포 단위. 자신만의 DB를 소유한다.
- **API Gateway**: 클라이언트의 단일 진입점. 라우팅·인증·요청 집계·속도 제한 등 횡단 관심사를 담당한다. → [API Gateway](./api-gateway.md)
- **서비스 디스커버리(Service Discovery)**: 동적으로 뜨고 사라지는 서비스의 위치(IP·포트)를 등록·조회하는 전화번호부. → [서비스 디스커버리](./service-discovery.md)
- **메시지 브로커(Message Broker)**: 서비스 간 비동기 통신을 중개하는 큐/스트림(Kafka, RabbitMQ 등). 발행-구독으로 느슨한 결합을 만든다. → [이벤트 기반](./event-driven.md)
- **분산 설정(Config Server)**: 여러 서비스의 설정을 한곳에서 관리하고 변경 시 전파한다.
- **안정성(Resilience) 인프라**: 한 서비스의 장애가 전체로 번지지 않도록 막는 회로 차단기·재시도·타임아웃. → [안정성](./resilience.md)

#### 서비스 분해 전략 (어떻게 나눌 것인가)
가장 흔한 실수는 **기술 계층(컨트롤러·서비스·DAO)** 으로 나누는 것이다. 이렇게 하면 기능 하나를 바꿔도 여러 서비스를 동시에 고쳐야 한다.
올바른 기준은 **비즈니스 능력(business capability)** 이다.

- **DDD(Domain-Driven Design, 도메인 주도 설계)**: 도메인 전문가의 언어(유비쿼터스 언어)로 모델을 설계한다.
- **바운디드 컨텍스트(Bounded Context)**: 하나의 모델이 일관되게 통용되는 경계. 같은 "상품"이라도 주문 컨텍스트의 상품과 배송 컨텍스트의 상품은 의미·속성이 다르다. **이 경계가 곧 서비스 분리선의 후보**가 된다.
- **비즈니스 능력 기준**: "회원 관리", "주문 처리", "결제", "배송" 처럼 조직이 제공하는 가치 단위로 나눈다. 변경 이유가 같은 것끼리 묶고(높은 응집), 변경 이유가 다른 것은 분리한다(낮은 결합).

```
[기술 계층으로 자르기 - 안티패턴]      [비즈니스 능력으로 자르기 - 권장]
  ┌──────────────┐                    ┌────────┐ ┌────────┐ ┌────────┐
  │ Controller   │                    │ 회원    │ │ 주문    │ │ 결제    │
  ├──────────────┤                    │(C+S+DB)│ │(C+S+DB)│ │(C+S+DB)│
  │ Service      │   기능 변경 시       └────────┘ └────────┘ └────────┘
  ├──────────────┤   3개 모두 수정       각 서비스가 한 도메인을
  │ DAO          │   → 결합도 폭발       세로로 온전히 소유
  └──────────────┘
```

#### 서비스 간 통신: 동기 vs 비동기
서비스끼리 협력하는 방식은 크게 두 가지이며, 상황에 맞게 섞어 쓴다.

| 구분 | 동기(Synchronous) | 비동기(Asynchronous) |
| --- | --- | --- |
| 대표 기술 | REST(HTTP), gRPC | 메시지 브로커(Kafka, RabbitMQ) |
| 흐름 | 요청 → 응답을 기다림(blocking) | 메시지 발행 후 즉시 반환(fire-and-forget) |
| 결합도 | 높음(상대가 살아 있어야 함) | 낮음(상대가 잠시 죽어도 메시지는 큐에 남음) |
| 일관성 | 즉시 응답·강한 일관성에 유리 | 최종 일관성(Eventual Consistency) |
| 장애 전파 | 호출 체인 따라 전파되기 쉬움 | 격리됨(브로커가 완충) |
| 적합한 경우 | 즉답이 필요한 조회·검증 | 알림·이력 적재·후속 작업 트리거 |

- 동기 REST는 직관적이지만 **호출 체인이 길어지면 한 곳의 지연이 전체 응답 지연으로 번진다**. 이를 막기 위해 [안정성](./resilience.md) 패턴(회로 차단기·타임아웃)이 필수다.
- gRPC는 HTTP/2 기반의 바이너리 프로토콜로, 서비스 간 내부 통신에서 REST보다 빠르고 효율적이다.
- 비동기 메시징은 결합도를 낮추고 탄력성을 높이지만, **흐름 추적이 어렵고 최종 일관성을 다뤄야 한다**. → [이벤트 기반](./event-driven.md)

#### 데이터 관리: Database per Service
MSA의 핵심 원칙 중 하나는 **각 서비스가 자기 DB만 소유하고, 다른 서비스의 DB에 직접 접근하지 않는다**는 것이다(Database per Service).
데이터가 필요하면 그 데이터를 소유한 서비스의 API를 호출하거나 이벤트로 받아온다.

- 장점: 서비스별로 DB 기술을 자유롭게 선택(주문은 RDB, 추천은 NoSQL)하고, 스키마를 독립적으로 변경할 수 있다.
- 대가: 여러 서비스에 걸친 트랜잭션을 **단일 DB 트랜잭션으로 묶을 수 없다.** 이로 인해 **분산 트랜잭션** 문제가 발생하며, Saga 패턴 등으로 해결한다. → [분산 데이터 일관성](./data-consistency.md)

#### 배포 토폴로지: 앱과 DB는 '세트'로 쪼개진다
모놀리식과 MSA의 가장 큰 물리적 차이는 **무엇이 몇 개로 뜨는가**다. 흔한 오해가 "MSA는 애플리케이션은 그대로 두고 DB만 나눈다"는 것인데, 실제로는 **앱과 DB가 한 세트로 함께 쪼개진다.** "회원 DB"는 혼자 떠 있는 게 아니라 항상 "회원 서비스(앱) + 회원 DB"가 짝을 이룬다.

```
[모놀리식] 앱 1 + DB 1
   웹 ──▶ 앱 1개(회원+상품+주문+결제 코드 한 덩어리) ──JDBC──▶ 단일 DB
          (트래픽 늘면 같은 앱을 통째로 복제 = 스케일아웃)

[MSA] 앱 N + DB N (세트로 분리)
   웹 ──▶ 회원 서비스(앱) ──JDBC──▶ 회원 DB
          상품 서비스(앱) ──JDBC──▶ 상품 DB
          주문 서비스(앱) ──JDBC──▶ 주문 DB
          결제 서비스(앱) ──JDBC──▶ 결제 DB
          (주문 트래픽만 늘면 주문 서비스만 복제 = 독립 확장)
```

| 구분 | 모놀리식 | MSA |
| --- | --- | --- |
| 배포 단위(앱) | 1개(통짜 산출물) | N개(도메인마다 독립 산출물) |
| 실행 인스턴스 | 같은 앱을 통째로 복제 | 서비스별로 독립 복제(독립 확장) |
| 논리 DB | 1개(전 도메인 한 곳) | N개(Database per Service) |
| 물리 DB 서버 | 보통 1대(+리플리카) | N대 또는 1대 안에 스키마 N개 |

- **앱도 함께 나뉜다**: 회원·상품·주문·결제가 각각 독립 배포되는 별도 자바 앱(별도 프로세스·포트)으로 뜬다. "앱은 1개고 DB만 N개"가 아니다. 이렇게 앱이 N개로 나뉘기 때문에 [저장소 전략](./repository-strategy.md)(멀티레포/모노레포) 논의가 따라온다.
- **물리 ≠ 논리**: "DB를 나눈다"는 건 **논리적 소유권 분리**다. 물리적으로는 한 DB 서버 안에 DB(스키마) N개로 둘 수도(소규모·학습용), 완전히 별도 서버 N대로 둘 수도 있다. 물리 배치가 어떻든 **"서로의 DB에 직접 접근하지 않는다"는 경계만 지키면** Database per Service 원칙은 충족된다.
- **계층은 단순하다**: `웹 → (API Gateway) → 서비스 앱 → DB 서버`. 서비스 앱이 자기 DB에 **직접** JDBC로 붙으며(모놀리식과 동일한 방식), 앱과 DB 사이에 "DB 전용 애플리케이션" 같은 중간 계층은 **없다**. 한 서비스 앱 안에 ①남의 API 요청을 받는 입구와 ②자기 DB에 SQL을 날리는 부분이 함께 들어 있다.

#### 서비스 간 데이터 조회: JOIN이 아니라 API 컴포지션
모놀리식에서는 여러 도메인 데이터가 한 DB에 있어 **SQL JOIN 한 번**으로 끝난다. MSA에서는 각 서비스가 **자기 DB에만 직접 접근**하므로 조회 방식이 근본적으로 달라진다.

```
[모놀리식]  DB가 합쳐줌 (SQL JOIN)
   SELECT o.id, m.name, p.title
   FROM orders o JOIN members m ON ... JOIN products p ON ...

[MSA]  앱이 합쳐줌 (API 컴포지션 = application-level join)
   주문 서비스
     1) 자기 DB(주문)에서 주문 조회 → memberId, productId 획득
     2) 회원 서비스 API 호출 → 이름
     3) 상품 서비스 API 호출 → 상품명
     4) 받아온 조각을 자바 코드에서 하나의 응답으로 조립
```

- **자기 DB만 접근(소유권 경계)**: 주문 서비스 코드에는 회원 DB 접속 정보(열쇠)조차 없다. 회원 데이터가 필요하면 회원 *서비스*의 API를 호출하고, 그 호출을 받은 회원 서비스가 자기 회원 DB를 조회해 응답한다. 즉 "DB를 본다"가 아니라 "그 데이터의 주인 서비스에게 물어본다".
- **cross-DB JOIN은 공유 DB 안티패턴**: 물리적으로 한 DB 서버에 함께 있으면 `order_db.orders JOIN member_db.members` 같은 교차 조인이 문법적으로는 가능하다. 하지만 이렇게 하면 회원 서비스의 스키마 변경이 주문 서비스 쿼리를 깨고, 회원 DB를 다른 서버·다른 DBMS로 옮기는 순간 불가능해진다. **물리적으로 가능하다 ≠ 해도 된다.** 경계는 물리 위치가 아니라 *소유권*으로 정의된다(아래 **주의점**의 '공유 데이터베이스' 안티패턴 참고).
- **호출 폭증의 대가**: API 컴포지션은 조회 한 번에 여러 서비스를 호출하므로 왕복이 늘어난다(N+1 호출). 조회가 잦은 화면은 이벤트로 필요한 데이터를 미리 한곳에 비정규화해 복제하는 **읽기 모델(CQRS)** 로 푼다. → [분산 데이터 일관성](./data-consistency.md)

#### 조회의 시작점 vs 데이터 소유자
같은 데이터 풀(회원·상품·주문·결제)을 다뤄도, **요청의 시작점이 어느 서비스냐에 따라 "직접 조회(자기 DB)"와 "REST로 받기(남의 서비스)"가 회전**한다.

| 진입(시작점) API | 시작점 서비스 | 자기 레포에서 직접 | 나머지는 REST 호출 |
| --- | --- | --- | --- |
| 회원 조회(+구매내역·상품·결제) | 회원 서비스 | 회원 정보 | 주문·상품·결제 |
| 상품 조회(+구매자·결제·상세) | 상품 서비스 | 상품 상세 | 주문·회원·결제 |
| 주문 조회(+사람·상품·결제) | 주문 서비스 | 주문 정보 | 회원·상품·결제 |

- 세 API는 **중복이 아니라 관점이 다른 조회**다. 시작점마다 "주인공(직접 조회)"과 "조연(REST 호출)"이 자리를 바꾼다.
- 주의: **시작점 ≠ 데이터 소유자**일 수 있다. "회원이 구매한 내역"의 실제 소유자는 주문 서비스이므로, 회원 서비스가 그 조회를 제공하려면 결국 주문 서비스를 호출해야 한다. 그래서 조회 API는 보통 **응답의 핵심(主) 데이터를 소유한 서비스**가 갖는 것이 자연스럽다.
- 조립 주체(조립자)는 요청마다 바뀐다. 보통은 시작점 서비스가 겸하지만, 한 화면에 여러 도메인이 섞이면 [API Gateway](./api-gateway.md)·[BFF](./bff.md) 같은 전용 조립 계층으로 분리한다.

### 장점 / 단점(트레이드오프)
MSA는 만능이 아니라 **명확한 트레이드오프를 가진 선택**이다.

- 장점
  - **독립 배포**: 서비스 단위로 배포해 배포 위험과 주기를 줄인다.
  - **독립 확장**: 부하가 큰 서비스만 골라서 스케일아웃 → 자원 효율 향상.
  - **기술 다양성(Polyglot)**: 서비스마다 가장 적합한 언어·DB·프레임워크 선택.
  - **장애 격리**: 한 서비스가 죽어도 나머지가 살아남도록 설계 가능.
  - **팀 자율성**: 작은 팀이 하나의 서비스를 온전히 소유해 빠르게 의사결정.
- 단점(분산 시스템의 비용)
  - **운영 복잡도 폭증**: 서비스 수만큼 배포·모니터링·로깅·추적 대상이 늘어난다.
  - **네트워크 비용·신뢰성 문제**: 메서드 호출이 네트워크 호출로 바뀌어 지연·실패가 일상이 된다.
  - **분산 데이터 일관성**: 트랜잭션을 한 번에 못 묶어 최종 일관성을 다뤄야 한다.
  - **테스트 난이도**: 통합·종단(E2E) 테스트가 여러 서비스를 띄워야 해 복잡하다.
  - **초기 비용**: 게이트웨이·디스커버리·CI/CD 등 인프라를 먼저 갖춰야 해 소규모 프로젝트에는 과하다.

> MSA는 "모놀리식이 너무 커져서 더 이상 감당이 안 될 때" 도입하는 것이 일반적이며,
> 처음부터 MSA로 시작하는 것보다 **모놀리식으로 시작해 경계가 분명해지면 점진적으로 쪼개는** 접근이 권장된다.

### 실무 예제
Java/Spring 진영에서는 **Spring Cloud** 생태계가 MSA 구성요소를 표준적으로 제공한다.

Ex: 쇼핑몰을 MSA로 구성할 때 각 인프라 요소는 다음과 같이 매핑된다.

| MSA 구성요소 | Spring Cloud 구현체 |
| --- | --- |
| 서비스 디스커버리 | Spring Cloud Netflix **Eureka** (서버/클라이언트) |
| API Gateway | **Spring Cloud Gateway** |
| 분산 설정 | Spring Cloud **Config Server** |
| 안정성(회로 차단기) | **Resilience4j** (Spring Cloud Circuit Breaker) |
| 서비스 간 동기 호출 | OpenFeign / RestClient |
| 비동기 메시징 | Spring Cloud Stream + Kafka/RabbitMQ |

Ex: 디스커버리(Eureka) 클라이언트가 되는 "주문 서비스"의 `application.yml` 예시.

```yaml
spring:
  application:
    name: order-service          # 디스커버리에 등록될 서비스 이름(논리 ID)
  config:
    import: "configserver:"       # Config Server에서 설정을 가져옴

eureka:
  client:
    service-url:
      defaultZone: http://localhost:8761/eureka  # 디스커버리 서버 주소
    register-with-eureka: true    # 나를 디스커버리에 등록
    fetch-registry: true          # 다른 서비스 목록을 주기적으로 조회

resilience4j:
  circuitbreaker:
    instances:
      paymentClient:              # 결제 서비스 호출용 회로 차단기
        sliding-window-size: 10
        failure-rate-threshold: 50  # 실패율 50% 초과 시 회로 OPEN
        wait-duration-in-open-state: 5s
```

Ex: 회로 차단기를 적용한 동기 호출(OpenFeign + Resilience4j) 코드 스니펫.

```java
@FeignClient(name = "payment-service")  // 디스커버리에 등록된 이름으로 호출(IP 직접 지정 X)
public interface PaymentClient {
    @PostMapping("/payments")
    PaymentResult pay(@RequestBody PaymentRequest request);
}

@Service
@RequiredArgsConstructor
public class OrderService {

    private final PaymentClient paymentClient;

    // 결제 서비스 장애 시 fallbackPay()로 우회하여 장애 전파를 차단
    @CircuitBreaker(name = "paymentClient", fallbackMethod = "fallbackPay")
    public PaymentResult requestPayment(PaymentRequest request) {
        return paymentClient.pay(request);
    }

    private PaymentResult fallbackPay(PaymentRequest request, Throwable t) {
        // 결제 보류 처리 후 추후 재시도 이벤트 발행 등
        return PaymentResult.pending(request.getOrderId());
    }
}
```

해당 빌드 의존성(Gradle)은 다음과 같이 구성한다.

```groovy
dependencies {
    implementation 'org.springframework.cloud:spring-cloud-starter-netflix-eureka-client'
    implementation 'org.springframework.cloud:spring-cloud-starter-openfeign'
    implementation 'org.springframework.cloud:spring-cloud-starter-circuitbreaker-resilience4j'
    implementation 'org.springframework.cloud:spring-cloud-starter-config'
}
```

### 주의점 / 안티패턴
MSA를 잘못 적용하면 **모놀리식의 단점은 그대로 둔 채 분산의 복잡도만 떠안는** 최악의 상태가 된다.

- **분산 모놀리식(Distributed Monolith)**: 서비스로 쪼갰지만 서로 너무 강하게 결합되어, 한 서비스를 바꾸면 다른 서비스도 함께 배포해야 하는 상태. 동기 호출 체인이 길고 공유 라이브러리·공유 스키마에 묶여 있으면 발생한다. **"따로 배포할 수 없다면 그건 마이크로서비스가 아니다."**
- **나노서비스(Nanoservice)**: 서비스를 너무 잘게 쪼개 서비스 하나가 함수 하나 수준이 되는 상태. 비즈니스 가치당 네트워크 호출·운영 오버헤드가 폭증해 성능과 관리성이 모두 나빠진다. 분리 기준은 "크기"가 아니라 "비즈니스 능력의 경계"여야 한다.
- **공유 데이터베이스(Shared Database)**: 여러 서비스가 하나의 DB·테이블을 직접 공유하는 것. 스키마 변경이 모든 서비스에 파급되어 독립 배포가 불가능해진다. **Database per Service 원칙을 깨는 가장 흔한 실수**다.
- **분산 트랜잭션을 2PC로 강제로 묶기**: 강한 일관성에 집착해 여러 서비스 DB를 2단계 커밋으로 묶으면 가용성·확장성이 무너진다. 대신 Saga·이벤트 기반 보상 트랜잭션으로 최종 일관성을 받아들이는 것이 정석이다. → [분산 데이터 일관성](./data-consistency.md)
- **횡단 인프라 없이 시작**: 서비스만 쪼개고 디스커버리·게이트웨이·중앙 로깅·분산 추적을 갖추지 않으면 장애 추적이 불가능해진다.
- **조직과 아키텍처의 불일치**: 콘웨이의 법칙(Conway's Law)대로, 팀 구조가 서비스 경계와 맞지 않으면 서비스 경계가 무너진다. 작은 자율 팀 단위로 서비스를 소유하게 해야 한다.
